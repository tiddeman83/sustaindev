# v0.1.x: What We Learned About Local Models For Prework

**Read this before you adopt SustainDev's local-tier workflow.**

This document distills six case studies and 72 hours of probe-script iteration into the lessons that matter most for new adopters. The case studies themselves are still the source of truth — `docs/measurement/case-study-01.md` through `docs/measurement/case-study-06.md` — but for someone deciding whether and how to use a local model in their development workflow, those six documents add up to roughly 16,000 words. This is the abridged version.

## The Matrix In One Look

The local-tier strong-fit / weak-fit matrix in `core/rules/model-routing.md` describes ten task classes. As of v0.1.x, six rows are empirically backed:

| Task class | Verdict | Why |
|-----------|---------|-----|
| Idea expansion: rough capture → structured task brief | **Strong fit** | Hero-workflow centerpiece. Bounded structured output from rich context. |
| Idea triage / classification | **Conditionally strong** | Fast, zero hallucinations. Needs RISKS.md and MAINTAINABILITY_NOTES in context, plus human spot-check on edge cases. |
| Codemap drafts from file lists | **Conditionally strong** | Accurate signatures and DI scope. Needs sufficient max_tokens (~8k for a 12-file bundle); reasoning consumes 50-70% of budget. |
| Summarization of long documents | **Conditionally strong** | High content quality. Packaging unreliable: at smaller context reasoning leaks into output; at larger context everything goes to reasoning_content channel. Recovery requires post-processing. |
| Full-document drafting from rich brief | **Weak fit** | Reasoning overhead and wall-clock cost outweigh cloud savings on a single artifact. Better routed to the cloud tier. |
| Risk extraction from existing prose | _(measured in case-study-07)_ | _(updated when v0.1.7 lands)_ |

The remaining four rows (adapter templates, cross-document consistency checks, time-critical authoring, batch/overnight prework) are still theoretical. The pattern across the six measured rows is consistent enough to predict them with reasonable confidence — but predicted is not measured.

## The Three Reasoning Regimes

This is the single most actionable insight from the v0.1.x case studies, surfaced most clearly in case-study-05 + case-study-06. The local model (Qwen 3.5 9B at MLX 4-bit on Apple Silicon at 16-24 GB) operates in three regimes depending on context size and output budget:

### Regime 1: Tight budget

When `max_tokens` is too low to hold both reasoning and structured output, **reasoning leaks into the visible content channel**. The output starts with "Thinking Process: 1. Analyze the request..." and the actual artifact gets truncated before it completes. The model knows the answer; the budget doesn't have room for both the answer and the thinking.

Evidence: case-study-05 Run 1 (4k cap on a 12-file catalog), case-study-06 Run 1 (2k cap on a 40KB summarization).

Failure mode: visible output polluted with planning prose; artifact incomplete.

### Regime 2: Generous budget at moderate context

When `max_tokens` is sufficient and `lm_studio_context` is sized just for the prompt, **reasoning channels properly to `reasoning_content`** and visible content is clean and complete.

Evidence: case-study-05 Run 2 (8k cap on a 12-file catalog at 16k context).

Failure mode: none structurally; the model produces a clean artifact with one minor flaw (a duplicate row) that the script's post-processing removes.

### Regime 3: Generous budget at large context

When `max_tokens` is sufficient but `lm_studio_context` is set substantially larger than the prompt needs, **the model invests all output in reasoning_content** and emits nothing in visible content.

Evidence: case-study-06 Run 2 (8k cap on a 40KB summarization at 32k context).

Failure mode: visible content empty; the script's defensive fallback ("if content is empty, use reasoning_content") rescues the data, but the user gets thousands of characters of "thinking out loud" instead of a clean artifact.

### The rule of thumb

**Set `max_tokens` just high enough to keep reasoning channeled. Set `lm_studio_context` just large enough for the prompt. Not larger.**

Larger contexts invite the model to think more, not to think more efficiently. This is counterintuitive but consistent across measurements. The probe scripts ship with sensible defaults (3000-4000 max_tokens for short tasks, 8000 for richer ones; user sets context to match prompt size). Don't override these to "use all my hardware."

## The Conditional-Strong-Fit Shape

Three of four measured strong-fit rows landed as **conditionally strong** rather than unconditionally. The conditions vary by task class, but the pattern is consistent: the underlying content is genuinely useful, while the packaging needs help.

This is significant because it changes how you should think about adopting the local tier. Don't ask "is the local model good enough?" — that's the wrong question. Ask "what's my packaging defense, and do I have time for the triage?"

The three conditions surfaced so far:

- **Triage condition.** For idea triage and codemap drafts, ~5-10% of the model's output needs a human spot-check. Plan for a triage step, sized to about 10-15 minutes per ~50 outputs. Without triage, you'll commit hallucinated test paths or miss high-risk files flagged for review.

- **Context condition.** For idea triage specifically, the model needs RISKS.md and MAINTAINABILITY_NOTES.md in its context, not just PROJECT_CONTEXT.md and CODEMAP.md. Without them, the model defaults to confident "commit-now" classifications and misses files that should be flagged for review.

- **Packaging condition.** For summarization, the script's post-processing must extract the artifact from wherever the model put it (visible content, reasoning_content channel, multiple drafts inside one response). v0.1.6 ships `scripts/lib/postprocess.py` to handle this; future probe scripts should import it from the start.

## The Post-Processing Pattern

Every probe script that asks the local model for structured output needs defensive post-processing. This isn't a quirk — it's an emergent property of how the model packages its output. Six categories of packaging quirk surfaced across the case studies, all encoded as utilities in `scripts/lib/postprocess.py`:

- **`<think>...</think>` tags inside content.** Strip with `strip_think_tags`.
- **Wrong arithmetic in summary lines.** The model can classify 80 files correctly but report `total=73` because counting drifts. Use `recompute_table_counts` to replace the model's claims with verified counts.
- **Duplicate rows in structured listings.** Use `dedupe_section_headers` to keep only the first occurrence of each named section.
- **Reasoning leaked into visible content as preamble.** Use `detect_planning_prose` and `extract_after_planning` to skip past the preamble.
- **Visible content empty with everything in reasoning_content.** Detect via `output == reasoning` and same length; treat reasoning as fallback content; then use `extract_last_draft` to pull out just the final structured artifact.
- **Indented markdown headings inside reasoning context.** `extract_after_planning` and `extract_last_draft` dedent based on the first matched line's indentation.

If you write a new probe script: import these from the start. Don't reimplement them inline. Don't trust the model's packaging.

## Setup Checklist For New Adopters

Before you run any probe script, set these up in this order:

1. **Install LM Studio.** [https://lmstudio.ai](https://lmstudio.ai)
2. **Download Qwen 3.5 9B at MLX 4-bit quantization** (recommended for Apple Silicon). Or Qwen 2.5 Coder 7B as fallback for older hardware. See `adapters/lm-studio/usage.md` for full options.
3. **Set context length to 16,384** (16k). The 4k default is too small for any serious project layer. Reload the model after changing the slider — the slider alone doesn't take effect until reload.
4. **Enable the Local Server** (LM Studio → Developer tab). Confirm with `curl http://127.0.0.1:1234/v1/models`.
5. **Author the project layer** (`PROJECT_CONTEXT.md`, `CODEMAP.md`, `AI_POLICY.md`, `MAINTAINABILITY_NOTES.md`, `VERIFY.md`, `RISKS.md`, `DECISIONS.md`). The probe scripts read these as model context. Without them, the local tier degrades to generic-LLM behavior — fast, but not project-aware.
6. **Run a probe in `--dry-run` first.** Always. The pre-flight check tells you whether your prompt fits in your context. If it doesn't fit, you'll get an actionable error before wasting a model call.

## What Worked, What Didn't, What Conditions

A summary table for the impatient:

| Question | Answer |
|----------|--------|
| Does the local tier produce zero hallucinations? | Yes, on signatures and structured fields. ~95-100% accuracy across the six case studies. Hallucinations were on specific facts (one wrong file path in case-study-03, "DI scope = singleton everywhere" overgeneralization in case-study-05) — never on signatures or method counts. |
| Can the local tier replace the cloud tier for code change? | No. That's not the role. Local is prework: idea expansion, triage, summarization, codemap drafting. Cloud is for writing code, full-document authoring, multi-file refactors. The router separates them on purpose. |
| Will the local tier save tokens? | Yes, on the prework step that would otherwise burn cloud tokens. ~2,500-7,500 cloud tokens per task converted to 0 cloud tokens. The trade-off is wall-clock: 2-5 min local instead of 30-60s cloud, plus 5-15 min human triage on the output. |
| Will the local tier save time? | Sometimes. Net wall-clock is roughly equivalent to cloud-only on a single task. The savings compound across many tasks (the cloud cost stays at zero; human triage is bounded). The real win is on tasks where the alternative is "Cowork-prepares-the-brief" at ~7-15 min — local can do that in 2-5 min. |
| Do I need to read the case studies? | Read this doc plus the matrix in `core/rules/model-routing.md`. The case studies are the audit trail for the claims; you don't need them for adoption. Read them when something surprises you or when adding a new probe script. |

## Where v0.1.x Doesn't Have Answers Yet

Honest acknowledgements of what we still don't know:

- Other model sizes (smaller 7B, larger 14B/22B/32B) were not tested. The matrix applies to Qwen 3.5 9B specifically. A larger model might shift weak-fit rows to conditionally-strong; a smaller one might fail on the strong-fit rows.
- Other hardware tiers (8GB Apple Silicon, discrete GPUs, x86 servers) were not tested. The 16-24 GB range is documented; below or above is not.
- Other source languages (English, Spanish, French) were not tested at scale. The probes worked on Dutch project material; consistency across languages is unverified.
- Other LM Studio versions (alternative quantizations, GGUF vs MLX, alternative inference engines) are not measured. MLX is recommended on Apple Silicon based on the speed/memory tradeoff; the case studies all used it.
- Multi-day usage is not measured. All the case studies are single-shot probes. Whether the local tier holds up across a full development week (60+ prework tasks) is the next question.

## What v0.2 Would Address

If we keep going, the priorities — derived from the case-study findings — are:

- Adapter expansion (Warp, VS Code, Cursor rules export, Cline rules) so the project is usable beyond LM Studio probe scripts.
- The remaining four matrix rows (adapter templates, cross-document consistency, time-critical authoring, batch/overnight) measured.
- A test suite for `scripts/lib/postprocess.py` so the consolidation can be relied on as a stable surface.
- Multi-day usage measurement: deploy SustainDev on one of the maintainer's real projects for a full week, log every probe invocation, write up the integrated experience as case-study-NN.
- Service catalog auto-update tooling (so it doesn't go stale as the project evolves).

None of these are urgent. The v0.1.x line stands on its own as a credible research artifact: six case studies, three probe scripts, one shared utility module, six tagged releases. It's a thesis with empirical backing — which is more than the same project had 72 hours ago.
