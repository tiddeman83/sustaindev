# Measurement Methodology

## Purpose

Measurement is absolutely mandatory because without explicit numbers, any claim regarding "sustainable" development is simply marketing. With precise numbers, workflow claims become evidence-backed realities. This file strictly standardizes exactly how to measure model usage, cost, and latency across the project. It outlines the specific metrics to capture but does not collect the measurements itself; case studies are explicitly responsible for capturing and presenting the actual data.

## What Gets Measured

- **Input tokens:** The exact number of tokens transmitted to the model in the prompt payload.
- **Output tokens:** The exact number of tokens generated by the model in the completion response.
- **Wall-clock seconds:** The strict end-to-end latency experienced by the user, measured from request dispatch to final artifact write.
- **Dollar cost:** The API cost incurred, calculated where applicable based on the provider's pricing tier at the exact time of the run.
- **Output quality assessment:** A qualitative review determining whether the output genuinely meets the requirements of the task.
- **Completeness:** A binary assessment defining whether the task was fully finished or only partially completed.

## What "A Task" Means

A task is fundamentally defined as a single, complete unit of work that demonstrates a clear, measurable before-and-after state. Examples include drafting a specific configuration file, classifying an isolated idea, or producing a comprehensive codemap. Multi-task workflows must be measured strictly by summing the individual per-task measurements. Continuous chat sessions are entirely excluded; they do not represent discrete tasks for measurement purposes and are considered purely noise.

## Comparing Approaches

The definitive standard comparison is strictly "approach A versus approach B for the exact same task." Both approaches must be run end-to-end using the exact same task definition and input context. Cherry-picking one approach's absolute best run against another approach's worst run is completely invalid; it functions as advocacy, not empirical measurement.

## Token Counting Caveats

Tokenizers fundamentally differ across various providers. OpenAI utilizes tiktoken, Anthropic leverages its own proprietary tokenizer, and local models typically use distinct local algorithms. Consequently, cross-provider token comparisons are purely approximate. Dollar cost remains the definitive apples-to-apples metric when attempting to compare efficiency across different API providers. Wall-clock latency stands as the definitive apples-to-apples metric when comparing the true user experience.

## What to Report

A formal measurement report, stored as `docs/measurement/case-study-XX.md`, must robustly include the following: a clear task description, the exact tool setup (specifying the cloud model, local model if applicable, and underlying hardware), a table detailing the per-approach metrics, a qualitative quality comparison, an assessment of what worked and what distinctly failed, and explicit reproducibility notes. The issue template located at `.github/ISSUE_TEMPLATE/measurement_report.md` perfectly mirrors this required structure to facilitate consistent community submissions.

## What Not to Claim

- Do not claim percentage savings on a single isolated task, as percentages hold actual meaning only when aggregated across many tasks of the exact same class.
- Do not extrapolate performance metrics from one specific hardware tier to an untested tier without performing the actual measurement.
- Do not present a rough estimate or theoretical projection as a definitive measurement; estimates must be clearly and explicitly labeled as such.
- Do not claim a specific workflow is broadly "10x faster" or generally "90% cheaper" without referencing a documented measurement that mathematically produced those exact numbers.

## How To Reproduce a Measurement

Every published case study must include sufficient, exacting detail—including precise model versions, the exact prompts used, hardware specifications, and configuration settings—so that an independent reproducer can attempt the exact same measurement and report whether they successfully obtained the exact same numbers. If a reported measurement fundamentally cannot be reproduced by an independent party, it immediately ceases to be evidence and is classified solely as an anecdote.

## Verification

The methodology itself is continually reviewed and systematically updated as diverse case studies accumulate over time. If multiple independent case studies effectively demonstrate that a specific measurement convention produces consistently misleading comparisons, the underlying methodology explicitly updates, and the existing historical case studies carefully note the specific methodology version they originally utilized.
