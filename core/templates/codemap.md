## Where Common Tasks Touch

- To add a new API endpoint, look in `<path-to-api-layer>/` and update the matching tests in `<path-to-api-tests>/`.
- To change the data model, look in `<path-to-models>/` and check migrations or serialization rules in `<path-to-data-boundary>/`.
- To update UI behavior, look in `<path-to-ui-components>/` and the state or view-model layer in `<path-to-ui-state>/`.
- To change validation, look in `<path-to-validation>/` and verify that error messages remain centralized.
- To update build configuration, look in `<build-config-file>` and `<ci-config-file>`.
- To add or update tests, look in `<path-to-tests>/` and mirror the structure of the code being changed.
- To change deployment behavior, look in `<path-to-deployment-config>/` and confirm the commands in `VERIFY.md`.

## Stable Anchors

- `<entry-point-file>` is the main runtime entry point and should be read before changing startup behavior.
- `<main-config-file>` defines project-wide configuration and should be changed deliberately.
- `<public-interface-file-or-dir>` exposes the boundary other modules depend on.
- `<dependency-manifest>` defines runtime and tooling dependencies.

## Hot Spots

- `<frequently-changed-path>` changes often because <short reason>; inspect nearby tests before editing.
- `<bug-prone-path>` receives many fixes because <short reason>; avoid broad refactors during feature work.
- `<integration-path>` changes when external services or file formats change; verify with realistic samples.
- `<legacy-path>` carries older conventions; follow existing behavior unless the task is explicitly a cleanup.

## What's Out of Scope

AI tools should generally avoid generated code, vendored dependencies, build artifacts, coverage output, and local cache directories. Test fixtures should only be read when the task is testing-specific or the fixture format is part of the behavior being changed. If a file is ignored by git or regenerated by the build, do not treat it as a source of truth.
