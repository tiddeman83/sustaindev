# Development Principles

## Purpose

This document is the foundation that every workflow and automated task in SustainDev rests on. It is a foundation, not a checklist. The principles relate to the rules in `token-efficiency.md`, `maintainability.md`, and `model-routing.md`, but they serve a different purpose: principles tell you why decisions are made; rules tell you what to do. When a situation falls outside the rules, the principles are the tie-breaker.

## The Principles

1. **Understand the local project before changing it.** Read the project context files, codemap, and existing structure before any change. Changes that ignore what is already there break things that are already working.
2. **Prefer narrow context over broad context.** Send the model the specific files and line ranges the task needs. Full repository scans dilute focus and waste tokens.
3. **Use codemaps and project context before scanning a whole repository.** Curated summaries locate files faster than brute-force search. Fall back to broad search only when the codemap is silent or stale.
4. **Preserve project architecture unless there is a concrete reason to change it.** Adapt to existing patterns rather than introducing new ones because they are familiar.
5. **Treat maintainability as a required output of every workflow.** Every code change must leave the system easier — or at least no harder — to read, test, and modify in the next session.
6. **Make verification explicit before implementation starts.** Define how you will prove the change works before you write the first line of implementation.
7. **Keep reusable instructions in the core layer, not scattered across tools.** Durable knowledge, policies, and workflows live in `core/` so they can be referenced from anywhere instead of duplicated.
8. **Keep adapters thin and generated where possible.** Tool-specific adapter files expose the core in the required format and do little more. Logic lives in `core/`.
9. **Route context to the appropriate source.** Use MCPs for external, dynamic, structured, or interactive context. Use local files and codemaps for stable, project-owned context.
10. **Capture new ideas separately from active implementation and prepare them immediately.** Extract unrelated ideas into their own documents while the context is fresh. The current task stays focused; the captured idea stays useful.
11. **Schedule non-urgent model-heavy work for better execution windows.** Push large structural tasks, batch prework, or slow verifications to offline queues. Live sessions are for latency-sensitive work.
12. **Require human review after unattended work.** Code generated by autonomous or scheduled runs is reviewed before it merges or deploys.
13. **Prefer fewer high-quality skills over many overlapping prompts.** Build and maintain a small set of effective workflows. A sprawling prompt collection is harder to maintain than the problems it solves.

## Applying the Principles

Principles inform decisions when rules do not cover the case. When two principles conflict, prefer the one that protects long-term maintainability. If adding a feature quickly with narrow context would break an architectural pattern, preserve the architecture. When a workflow is ambiguous or a new tool integration is proposed without precedent, the principles point to the safer default. The goal is a sustainable codebase over time, not temporary velocity at the expense of future clarity.

## Verification

Principles are not directly testable. Their effect shows up in the maintainability of the repository over months and in token-cost efficiency measured over weeks. Evidence accumulates in `docs/measurement/` as case studies record the cost, speed, and quality impact of workflows built on this foundation.
