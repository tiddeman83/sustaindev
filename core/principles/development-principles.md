# Development Principles

## Purpose

This document provides the foundational principles that govern every workflow and automated task in SustainDev. It functions as a foundation, not a checklist. These principles relate directly to the operational rules found in `token-efficiency.md`, `maintainability.md`, and `model-routing.md`, but they serve a distinctly different purpose: principles tell you why decisions are made, while rules tell you what to do in specific scenarios. When you face an ambiguous situation where specific rules do not cleanly apply, returning to these principles ensures consistency with the project's long-term goals. They form the underlying rationale that makes the strict rules necessary.

## The Principles

1. **Understand the local project before changing it.** Never modify a codebase without first reading its context files, codemaps, and existing structure to ensure your changes align perfectly with what is already there.
2. **Prefer narrow context over broad context.** Supply models only with the specific files and line ranges needed for the task, rather than flooding the context window with full repository scans that dilute focus and waste tokens.
3. **Use codemaps and project context before scanning a whole repository.** Rely heavily on curated architectural summaries to locate files and understand boundaries, falling back to brute-force searching only when the codemap is stale or silent.
4. **Preserve project architecture unless there is a concrete reason to change it.** Adapt rigorously to the existing patterns, conventions, and structures rather than introducing new paradigms simply because they are familiar or momentarily convenient.
5. **Treat maintainability as a required output of every workflow.** Every single code change must leave the system in a state where it remains easy to read, test, and modify for the next session or the next developer.
6. **Make verification explicit before implementation starts.** Define exactly how you will test and prove that a code change works before you write the very first line of implementation logic.
7. **Keep reusable instructions in the core layer, not scattered across tools.** Centralize durable project knowledge, policies, and workflows in a tool-neutral location (`core/`) so they can be consistently referenced everywhere.
8. **Keep adapters thin and generated where possible.** Ensure that tool-specific adapter files do little more than expose the core instructions in the required format, entirely avoiding logic duplication.
9. **Route context to the appropriate source.** Use MCPs strictly for external, dynamic, structured, or interactive context, while relying on local files and codemaps for stable, project-owned context.
10. **Capture new ideas separately from active implementation and prepare them immediately.** Extract unrelated ideas or deferred tasks into separate documents while the context is still fresh, allowing the current implementation to remain intensely focused.
11. **Schedule non-urgent model-heavy work for better execution windows.** Push large structural tasks, batch prework, or slow verifications to offline queues rather than blocking active, latency-sensitive development sessions.
12. **Require human review after unattended work.** Never merge or deploy code generated by autonomous or scheduled runs without a human developer thoroughly verifying the outcome and impact.
13. **Prefer fewer high-quality skills over many overlapping prompts.** Build, measure, and meticulously maintain a small set of highly effective, targeted workflows rather than accumulating a sprawling collection of undocumented prompt fragments.

## Applying the Principles

Principles exist to inform decisions when concrete rules do not cover a specific case or edge condition. They provide the definitive tie-breaker for structural dilemmas. When two principles appear to conflict, always prefer the one that protects long-term maintainability. For example, if adding a feature quickly using narrow context risks breaking an established architectural pattern, the decision must heavily favor preserving the architecture. When a workflow is ambiguous, or a new tool integration is proposed without clear precedent, these principles point to the safer, more durable default. The overarching goal is always a predictable, sustainable codebase over the long haul, never temporary velocity at the expense of future clarity.

## Verification

Principles are not directly testable in the way that syntax validation or unit tests are. Instead, their true effect shows up in the maintainability of the repository over months and in the token cost efficiency measured over weeks. The concrete evidence that these principles are working as intended accumulates in the `docs/measurement/` directory, where detailed case studies record the empirical cost, speed, and quality impact of workflows designed strictly around this foundation.
